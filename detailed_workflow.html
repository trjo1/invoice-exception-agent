Step 1PDF upload — what happens server-side

Pre-LLM plumbing. No model calls; this step is just FastAPI handling the file upload, persisting it, and creating a run row so we can show streaming progress on the next page.

The request

The /demo upload form POSTs multipart/form-data to /demo/run-streaming:

POST /demo/run-streaming HTTP/1.1
Content-Type: multipart/form-data; boundary=----...

------...
Content-Disposition: form-data; name="file"; filename="P004_idx0428.pdf"
Content-Type: application/pdf

%PDF-1.7
... binary bytes ...
%%EOF

What the handler does

The route handler in src/p2p_agent/hitl/webapp/server.py:

  1. Validates that the filename ends in .pdf. Otherwise returns 400.
  2. Reserves a run ID by calling PipelineRunStore.create(). This inserts a row in the pipeline_runs SQLite table with status='running' and a UUID. Used as both the URL and the on-disk filename so we don't have to read the file twice.
  3. Streams the bytes to disk in 64 KB chunks to logs/demo_uploads/{run_id}.pdf, enforcing a 10 MB cap.
  4. Returns a 303 redirect to /demo/run-stream-view/{run_id} — the live timeline page that opens an SSE connection to start the pipeline.

Important: this endpoint does not run the pipeline. The pipeline runs inside the SSE handler so the user sees progress streamed instead of waiting on a blocking POST for 30-200s.

Persistence
Three things now exist on disk: (a) the PDF at logs/demo_uploads/{run_id}.pdf, (b) a row in logs/hitl_queue.db::pipeline_runs with status=running, and (c) the file path stored on that row. Any failure from here on can be diagnosed by re-opening /demo/run/{run_id} — the run is durably tracked.

Step 2PDF → text (pypdf)

Pure-Python text-layer extraction. Runs before the LLM is called. The LLM sees the text output, not the PDF bytes — which is why scanned/image-only PDFs aren't supported yet (vision path would swap in here).

Implementation is _read_pdf_text() in src/p2p_agent/extractors/invoice_extractor.py:

def _read_pdf_text(pdf_path: Path) -> str:
    reader = PdfReader(str(pdf_path))
    pages: list[str] = []
    for page in reader.pages:
        pages.append(page.extract_text() or "")
    return "\n\n".join(pages)

The system prompt — full text

Loaded from src/p2p_agent/llm/prompts/invoice_extraction.md. This is what ships to the model on every extract call:

You are the invoice-extraction node of a procure-to-pay (P2P) AI agent. You receive the text of one invoice PDF and your job is to extract the structured fields verbatim — including any typos, missing values, or odd formatting that's on the page.

# Core principle: extract, don't correct

- Output what's on the page, not what you think should be there.
- If the PO reference looks typo'd (`PO-2026-04-04134` when you suspect `00134`), output `PO-2026-04-04134` exactly.
- If a tax line is missing, leave the `tax` array empty. Do NOT invent a tax line.
- If a field is present but illegible / unclear, output an empty string and lower the confidence.
- Correction logic belongs in downstream nodes (classifier, PO matcher). Yours is the eyes.

# Output schema

Return JSON only. No prose outside the JSON. No chain-of-thought. Wrap the JSON in a ```json code block.

```json
{
  "invoice_id": "<the supplier's invoice number printed on the page>",
  "po_reference": "<PO number as printed; empty string if absent>",
  "invoice_date": "<YYYY-MM-DD; normalize to ISO format>",
  "currency": "<ISO 4217 code, e.g. USD / EUR / INR / BRL / GBP>",
  "payment_terms": "<e.g. NET-30, NET-45, 2/10-NET-30, DUE-ON-RECEIPT>",
  "header_fields": {
    "vendor_name": "...",
    "vendor_address": "...",
    "vendor_tax_id": "...",
    "buyer_name": "...",
    "buyer_address": "...",
    "buyer_po_contact": "..."
  },
  "line_items": [
    {"line_no": 1, "sku": "...", "description": "...", "quantity": <n>, "unit_price": <n>, "line_total": <n>}
  ],
  "subtotal": <n>,

  "total": <n>,
  "field_confidence": {"po_reference": <0-1>, ...}
}
```

# Field-confidence guidelines

- **0.95+** — Field is clearly printed, unambiguous.
- **0.80–0.94** — Field clear but minor formatting oddity.
- **0.60–0.79** — Field present but unclear; OCR-style ambiguity.
- **Under 0.60** — Illegible, missing, or guessed.

# Hard rules

- All amounts as plain JSON numbers — no currency symbols, no thousands separators.
- `line_total = quantity × unit_price` (within rounding). Don't override what's printed.
- `subtotal = sum of line_totals`. `total = subtotal + sum(tax.amount)`.
- If a number on the page violates these identities, output what's on the page anyway. The classifier will flag it.
- Symbols: € → EUR, £ → GBP, ¥ → JPY, $ → USD (default), R$ → BRL, ₹ → INR.
- Dates: ISO 8601. Ambiguous dates use vendor's regional convention, lower confidence to ~0.7.

Step 4Cross-case context — six local lookups (no LLM)

After extraction, the agent enriches the invoice with master-data lookups before asking the classifier anything. This is what turns "isolated invoice" into "case in context." No LLM calls here — everything is sub-100ms local dict lookups against six JSON tables. In production these tables become SAP OData endpoints; the interface stays the same.

The six tables — what they look like

TableFileIndexed byWhy we look it up
Vendor master vendor_master.json tax_id, name, ID "Is this supplier known? What tier? Sanctions OK?"
PO master po_master.json PO ID "What did we authorize — currency, terms, line items, approvers?"
Goods receipts goods_receipts.json PO ID "Did we physically receive the goods?"
Payment status payment_status.json PO ID "How much of this PO has been invoiced + paid already?"
Invoice history invoice_history.json supplier invoice #, PO ID "Have we seen this exact invoice before? Other invoices on same PO?"
Vendor changes vendor_changes.json vendor ID "Did this vendor's bank/address/tax-id change recently?"

All six lookups run in parallel via asyncio.gather in CaseContextBuilder.build_async() — they're sub-millisecond against the JSON mocks today, but the parallelism pays off when these become real SAP calls (0.5–2s each).

1. Vendor master

The trickiest lookup because invoices print tax IDs inconsistently ("EIN: 12-3456789" vs "12-3456789" vs "123456789"). The lookup tries a four-stage resolution chain before giving up.

Table schema

{
  "by_id":              { "<vendor_id>": { VendorRecord } },
  "by_tax_id":          { "<normalized_tax_id>": { VendorRecord } },
  "by_tax_id_aggressive": { "<alphanum_lowercase>": { VendorRecord } },
  "by_tax_id_last6":    { "<last_6_digits>": { VendorRecord } },
  "by_name_tokens":     [ { "tokens": ["mumbai","materials"], "vendor": {...} }, ... ]
}

// Each VendorRecord:
{
  "id": "VEN-P004",
  "name": "Mumbai Materials Co.",
  "tax_id": "27AABCS1234B1Z2",
  "country": "IN",
  "addresses": ["..."],
  "bank_account_last4": "7124",
  "contract_type": "msa",          // msa | spot | recurring
  "tier": "strategic",             // strategic | tactical | new | inactive
  "onboarding_date": "2024-01-15",
  "status": "active",              // active | inactive | blocked
  "sanctions_check_passed": true,
  "phi_access": false,             // HIPAA flag
  "sbe_classification": ""

Lookup resolution chain — VendorMasterLookup.get_by_tax_id()

  1. Stage 1 — exact normalized tax_id. Strip "EIN:" / "VAT:" / "GSTIN:" / "PAN:" / etc. prefixes, then look up in by_tax_id.
  2. Stage 2 — aggressively normalized. Reduce to alphanumeric-lowercase only ("12-3456789" → "123456789", "DE 123 456 789" → "de123456789"). Looks up in by_tax_id_aggressive.
  3. Stage 3 — last 6 digits. Pull the last 6 numeric characters; look up in by_tax_id_last6. Catches reformatted tax IDs.
  4. Stage 4 — fuzzy name fallback. If tax_id lookup fails entirely, call get_by_name(). Strips accents (Müller → Muller, São → Sao), tokenizes (drops "Inc", "GmbH", "Ltd", etc.), and Jaccard-matches the token sets. Default threshold 0.6.

How smart signals are computed

The classifier doesn't read the raw CaseContext — it reads a curated list of one-line "smart signals" that come out of CaseContext.summary_signals(). This is the agent's interpretation of the lookups, with heuristic filters to suppress false positives.

The signal-generation logic — verbatim from the model

def summary_signals(self) -> list[str]:
    signals = []

    # Always emit an informational anchor for vendor + PO + GR.
    signals.append(f"vendor in master: {self.vendor_record.name} ..."
                   if self.vendor_record else "vendor NOT in master file")
    signals.append(f"PO {self.po_record.id} status={...}"
                   if self.po_record else "PO NOT found")
    signals.append(f"GR {self.goods_receipt.id} on file"
                   if self.goods_receipt else "no goods receipt recorded")

    # DUPLICATE: only fire when a prior invoice has same supplier_invoice_number
    # AND total within 5% of this invoice's total (a real duplicate, not just an ID collision).
    if self.prior_invoices_same_supplier_number:
        real_dups = [p for p in self.prior_invoices_same_supplier_number
                     if abs(p.total - this_total) / max(this_total, 1.0) < 0.05]
        if real_dups:
            signals.append(f"DUPLICATE: {len(real_dups)} prior invoice(s) with "
                           f"same supplier_invoice_number AND matching total")

    # SPLIT WATCH: only fire when ≥3 priors on same PO summing to ≥60% of PO authorization
    # (the classic split-invoice fraud signature; below this it's just multi-shipment billing).
    if len(self.prior_invoices_same_po) >= 3 and self.po_record:
        cumulative = sum(p.total for p in self.prior_invoices_same_po)
        if cumulative >= 0.6 * self.po_record.total_authorized:
            signals.append(f"SPLIT WATCH: {len(self.prior_invoices_same_po)} prior "
                           f"invoices on same PO totaling {cumulative} vs PO "
                           f"authorization {self.po_record.total_authorized}")

    if self.po_payment_status and self.po_payment_status.fully_paid:
        signals.append("PO ALREADY FULLY PAID")
    if self.vendor_recent_changes:

Why the smart-filter matters
The raw lookup found one prior invoice with the same supplier_invoice_number (P004_idx0228). But naively flagging that as a duplicate would be wrong — the priors had different totals (40769 vs 12390). The smart filter requires same supplier # AND total within 5% before firing DUPLICATE. The confidence guardrail (post-classify)

After parsing, one safety check runs in _apply_confidence_guardrail():

  1. If the model returned none → return as-is.
  2. If confidence ≥ 0.70 → return as-is.
  3. If confidence < 0.70 AND no actionable smart signal fired → downgrade to none with a note. This catches the residual "rich context made me hallucinate an exception" failure mode.

For our example, confidence is 0.95 → no downgrade. The classification is final.

What gets passed forward

Both the classification AND the case context flow forward. The retrieve step (next) uses the classification's category + evidence to query the policy library. The decide step (LLM call #3, the reasoning model) re-reads everything and picks the actual action.

Step 6Policy retrieval — RAG, no LLM call

Before the decide step runs, the agent retrieves the most relevant policy snippets from the buyer's policy library. This is pure RAG: embed the query, cosine-search 75 pre-embedded policy vectors, return the top-K. No LLM call. Sub-500ms after the embedder is warm.

Why retrieve instead of stuffing all 75 policies into the decide prompt?

Two reasons. First, the policy library scales — for a real buyer it's hundreds to thousands of policy clauses, not 75. Second, the decide model (R1) is slow; sending 50 KB of policy text on every call would 5× the prompt token cost. RAG keeps only the relevant 5 snippets in scope.

Query construction

The query string is built in _build_retrieval_query() from the classification + key invoice signals — not from raw text. This keeps the retrieval focused on the agent's understanding, not boilerplate.

def _build_retrieval_query(extraction_payload: dict, classification) -> str:
    parts = [
        f"Exception category: {classification.class_label.value}",
        f"Rationale: {classification.rationale}",
    ]
    if extraction_payload.get("currency"):
        parts.append(f"Currency: {extraction_payload['currency']}")
    parts.append("PO reference present: yes" if extraction_payload.get("po_reference")
                 else "PO reference missing")
    if classification.evidence:
        parts.append("Evidence: " + ", ".join(classification.evidence))
    return "\n".join(parts)

Step 8HITL routing — rules-based, no LLM call

The router converts a RecommendedAction into a RoutingDecision: which tier (1/2/3) and which named role (buyer / VP finance / fraud team / treasury / etc.). Pure rule-based — no LLM. Same input always produces the same routing. Sub-millisecond.

Why rules and not an LLM

Routing decisions are the boundary where the agent's authority ends and a human's begins. They have to be auditable, deterministic, and policy-driven — not "the model picked Tier 2 today." A spreadsheet-style rule table fits that constraint exactly.

Important: nothing has been sent yet
This draft is a string in memory. It will not reach the Fraud team, the supplier, the AP supervisor, or anyone else until a human reviewer hits Approve in step 9. That's the boundary — the agent never auto-sends. Tier 1 auto-pass is the only case where the system acts without a human, and Tier 1 never produces a draft to begin with.

Step 10aHuman-in-the-loop approval

For Tier 2 and Tier 3 cases, the agent stops here. The case lands in a SQLite-backed approval queue at /queue and waits for a human to approve, edit-and-approve, or reject. Every action is logged to an immutable audit trail.

What gets persisted

The pipeline enqueues a HITLItem row containing the full pipeline result (classification, recommendation, routing decision, draft) plus metadata. From hitl/queue.py:

HITLItem(
  id                = uuid4(),                  # primary key
  case_id           = pipeline_run_id,
  created_at        = now,
  tier              = routing_decision.tier,    # 2 or 3 (1 doesn't enqueue)

  routing_reason    = routing_decision.reason,

The three approval paths

PathWhat the reviewer doesNext step
Approve Hits "Approve" — endorses the agent's recommendation as-is. Status → approved; executor fires the recipe for the original action.
Edit and approve Modifies the draft (subject / body / recipient), then approves. Status → edited_approved; executor uses the edited draft on email steps; original draft preserved in audit log.
Reject Decides the agent got it wrong. Provides a note. Status → rejected; executor does NOT fire. Case closes; audit log captures the reviewer's note.

The audit trail

Every state change writes an immutable row to hitl_audit_entries with timestamp, actor, action ("approve" / "reject" / "edit_and_approve"), and note. The original draft and the edited draft are both preserved — there's no "lose track of what the agent originally said." This is the part compliance teams test in pilot.

Step 10bAction execution — SAP / Email / ServiceNow / PagerDuty

Once approved, the executor turns the recommendation into actual side effects. Today this is mock mode (every call is logged as "would have done X" but no real HTTP is fired). The real backend swap is gated on SAP credentials but uses the same execute(item) signature, so the swap is a backend change without any pipeline touch-up.

What an ExecutionStep looks like

class ExecutionStep(BaseModel):
    system:  str                              # SAP S/4HANA Cloud | Email | Slack |
                                              # ServiceNow | PagerDuty | Treasury | Internal
    verb:    str                              # POST | PUT | EMAIL | NOTIFY |
                                              # CREATE_TICKET | HALT_PAY_RUN | SCHEDULE_RECHECK
    target:  str                              # endpoint, recipient, channel, ...
    payload_summary: dict[str, Any]           # compact dict the UI can render

class ExecutionResult(BaseModel):
    status:      ExecutionStatus              # simulated_success | simulated_failed | skipped | executed
    steps:       list[ExecutionStep]
    note:        str
    executed_at: datetime

Where the edited-draft path matters

If the reviewer used "Edit and approve" on a step that emits an email step (e.g. request_supplier_credit_memo), _build_email_step() reads item.edited_draft_json first and falls back to payload_json.draft only if no edit was made. That's how a reviewer's word choice ends up on the actual outbound email instead of the original LLM-written body.

End of the trace
Every step from upload (1) → execute (10b) is now persisted to disk. Re-opening /demo/run/{run_id} at any later time reconstructs the entire trace from pipeline_runs.result_json + hitl_items.* + hitl_audit_entries.* + logs/llm_calls.jsonl. Nothing is held in memory. The agent is fully auditable: the buyer's compliance team can replay every decision after the fact, see which model returned what, see what the reviewer changed, and see the executor's recipe and outcome.
